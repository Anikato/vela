## 一、项目概述

**项目名称**：Vela（拉丁语"帆"，寓意全球贸易、乘风出海）

**定位**：一套自研的多语言展示型网站系统，初始场景为 B2B 外贸独立站，架构上可适配企业官网、产品目录站、品牌展示站等多种类型。

**核心设计原则**：

| 原则 | 含义 |
|------|------|
| **AI 优先开发** | 代码结构高度模式化，每个模块遵循相同的四层模式，方便 AI 理解和复制 |
| **类型贯通** | 从数据库 Schema 到 UI 组件，TypeScript 类型一路贯穿，零手写 DTO |
| **约定大于配置** | 文件命名、函数命名、目录结构全部有明确约定，减少决策点 |
| **渐进式扩展** | 核心功能先做，区块/模块/主题按需扩展，不过度设计 |

## 四、架构分层与开发模式

### 4.1 四层架构

```
┌─────────────────────────────────────────────┐
│               UI Layer (展示层)              │
│     Server Components / Client Components    │
│         网站前端 + 后台管理界面               │
├─────────────────────────────────────────────┤
│             Action Layer (动作层)             │
│            Server Actions / API Routes       │
│     输入校验（Zod）→ 调用 Service → 返回结果  │
├─────────────────────────────────────────────┤
│            Service Layer (业务层)             │
│              纯业务逻辑函数                   │
│     不依赖 Next.js，可独立测试和复用           │
├─────────────────────────────────────────────┤
│              Data Layer (数据层)              │
│           Drizzle ORM + PostgreSQL           │
│        Schema 定义 + 查询 + 迁移              │
└─────────────────────────────────────────────┘
```

### 4.2 各层职责边界

| 层 | 职责 | 不做的事 |
|----|------|---------|
| **UI** | 渲染界面、处理用户交互、调用 Action | 不直接访问数据库，不含业务逻辑 |
| **Action** | 接收 UI 请求、Zod 校验输入、调用 Service、格式化返回 | 不含复杂业务逻辑，不直接写 SQL |
| **Service** | 业务逻辑、数据组装、事务管理、调用外部 API（翻译/邮件） | 不依赖 Next.js API，不做输入校验 |
| **Data** | 表结构定义、数据库连接、迁移 | 不含业务逻辑 |

### 4.3 开发一个模块的标准流程

以"新闻管理"模块为例：

**Step 1：定义 Schema**

```typescript
// server/db/schema/news.ts
import { pgTable, uuid, varchar, text, timestamp, integer } from 'drizzle-orm/pg-core';
import { media } from './media';
import { languages } from './languages';

export const news = pgTable('news', {
  id: uuid('id').primaryKey().defaultRandom(),
  slug: varchar('slug', { length: 255 }).unique().notNull(),
  coverImageId: uuid('cover_image_id').references(() => media.id),
  status: varchar('status', { length: 20 }).default('draft').notNull(), // draft | published
  publishedAt: timestamp('published_at'),
  sortOrder: integer('sort_order').default(0).notNull(),
  createdAt: timestamp('created_at').defaultNow().notNull(),
  updatedAt: timestamp('updated_at').defaultNow().notNull(),
});

export const newsTranslations = pgTable('news_translations', {
  id: uuid('id').primaryKey().defaultRandom(),
  newsId: uuid('news_id').references(() => news.id, { onDelete: 'cascade' }).notNull(),
  locale: varchar('locale', { length: 10 }).references(() => languages.code).notNull(),
  title: varchar('title', { length: 500 }),
  summary: text('summary'),
  content: text('content'),           // 富文本 HTML
  seoTitle: varchar('seo_title', { length: 200 }),
  seoDescription: text('seo_description'),
}, (table) => ({
  uniqueNewsLocale: unique().on(table.newsId, table.locale),
}));
```

**Step 3：编写 Action**

```typescript
// server/actions/news.actions.ts
'use server';

import { z } from 'zod';
import * as newsService from '@/server/services/news.service';

const createNewsSchema = z.object({
  slug: z.string().min(1).regex(/^[a-z0-9-]+$/),
  status: z.enum(['draft', 'published']),
  coverImageId: z.string().uuid().nullable(),
  translations: z.record(z.string(), z.object({
    title: z.string().optional(),
    summary: z.string().optional(),
    content: z.string().optional(),
    seoTitle: z.string().optional(),
    seoDescription: z.string().optional(),
  })),
});

export async function createNewsAction(input: z.infer<typeof createNewsSchema>) {
  const parsed = createNewsSchema.safeParse(input);
  if (!parsed.success) {
    return { success: false, error: parsed.error.flatten() };
  }
  try {
    const result = await newsService.createNews(parsed.data);
    return { success: true, data: result };
  } catch (error) {
    return { success: false, error: 'Failed to create news' };
  }
}
```

### 4.4 Server Action 返回值规范

所有 Server Action 统一返回格式：

```typescript
// 成功
{ success: true, data: T }

// 失败（校验错误）
{ success: false, error: ZodFlattenedError }

// 失败（业务错误）
{ success: false, error: string }
```

类型定义：

```typescript
type ActionResult<T> =
  | { success: true; data: T }
  | { success: false; error: string | Record<string, string[]> };
```

### 4.5 数据查询模式

**后台（写操作）**：UI → Server Action → Service → DB

**网站前端（读操作）**：Server Component 直接调用 Service（无需经过 Action 层）


## 五、数据库设计

### 5.1 设计原则

| 原则 | 说明 |
|------|------|
| **翻译分表** | 每个可翻译实体有独立的 `_translations` 表，通过 `(entity_id, locale)` 唯一约束 |
| **UUID 主键** | 所有表使用 UUID v4，避免自增 ID 的安全隐患和分布式问题 |
| **软删除可选** | 核心实体（产品、分类）用状态字段（archived），不做物理删除；辅助数据可直接删除 |
| **JSON 灵活字段** | 区块配置、主题配置、模板配置使用 JSON 字段，避免频繁加列 |
| **级联删除翻译** | 主实体删除时，翻译表通过 `ON DELETE CASCADE` 自动清理 |
| **时间戳** | 所有表有 `created_at`，可变实体有 `updated_at` |

## 六、多语言架构

### 6.1 语言检测与路由中间件

```typescript
// src/middleware.ts 核心逻辑
// 1. 读取 URL 中的 locale 前缀
// 2. 如果无前缀 → 使用默认语言（不重定向）
// 3. 如果前缀 = 默认语言 → 308 重定向到无前缀版本
// 4. 如果前缀 = 有效的非默认语言 → 正常通过
// 5. 如果前缀 = 未知语言代码 → 判断为自定义页面 slug（见第 8 步）
// 6. 有 locale Cookie（非默认语言） → 302 重定向到 /{locale}/{path}
// 7. 将 locale 写入 Cookie（有效期 1 年）
// 8. 自定义页面 slug rewrite（透明，浏览器 URL 不变）：
//    - 默认语言：单段路径 /{slug} 不在静态路由表中 → 内部 rewrite 到 /page/{slug}
//    - 非默认语言：/{locale}/{slug} 第二段不在静态路由表中 → 内部 rewrite 到 /{locale}/page/{slug}
//    静态路由表（STATIC_FRONTEND_ROUTES）：products | news | about | contact | search | page
//    新增前台固定路由时，需同步将路径段加入此集合，避免被误判为自定义页面
```

## 七、区块系统架构

### 7.1 核心概念

```
注册表 (Registry)
  ├── 'hero'         → HeroSection 组件
  ├── 'rich_text'    → RichTextSection 组件
  ├── 'cta'          → CTASection 组件
  └── ... 按需注册

渲染调度器 (SectionRenderer)
  接收 sections[] 数组
  ├── 遍历每个 section
  ├── 查注册表找到对应组件
  ├── 做翻译回退（传入当前语言的翻译）
  └── 用 SectionWrapper 包裹（处理通用配置）
```

### 8.3 后台管理界面（Admin UI）规范

后台管理界面采用**暗色科技感主题**，独立于网站前台的主题系统。后台固定中文，无需多语言支持。

#### 主题机制

- 后台使用 `.admin-dark` CSS 作用域，在 `globals.css` 中定义全套 CSS 变量（深蓝黑背景 + 青色主色调）
- 通过 `useAdminTheme()` hook 将 `dark` + `admin-dark` 类挂载到 `<html>` 元素，确保 Radix Portal 弹出内容（Dialog、AlertDialog、DropdownMenu、Select、Sheet、Tooltip、Toast）也能继承后台主题变量
- 组件卸载时自动移除类名，不影响网站前台
- `dark:` Tailwind 变体在后台中可正常生效（因为 `<html>` 上有 `.dark` 类）
- 所有 shadcn/ui 组件在后台中使用原始 CSS 变量语义色（`bg-background`、`text-foreground`、`bg-card`、`bg-popover` 等），**不需要**手动在弹窗组件上添加 `admin-dark` 类

#### 色彩体系

| 语义 | 用途 | OKLCH 值 |
|------|------|----------|
| `--background` | 页面背景 | `oklch(0.105 0.012 260)` — 深蓝黑 |
| `--foreground` | 主文本 | `oklch(0.93 0.005 260)` — 淡蓝白 |
| `--card` / `--popover` | 卡片 / 弹出层背景 | `oklch(0.155 0.014 260)` — 比背景稍亮 |
| `--primary` | 主色调（按钮、高亮） | `oklch(0.75 0.14 200)` — 青色 |
| `--primary-foreground` | 主色上的文字 | `oklch(0.12 0.02 240)` — 深色 |
| `--muted` / `--secondary` | 次要区域 | `oklch(0.20 0.015 260)` |
| `--muted-foreground` | 次要文本 | `oklch(0.60 0.01 260)` |
| `--accent` | 悬停高亮 | `oklch(0.22 0.018 260)` |
| `--border` | 边框 | `oklch(0.30 0.02 260)` — 微光边框 |
| `--destructive` | 危险操作 | `oklch(0.65 0.22 25)` — 红色 |
| `--sidebar` | 侧边栏背景 | `oklch(0.09 0.015 260)` — 最深 |
| `--ring` | 焦点环 | 与 primary 一致 |

#### 组件使用规范

1. **弹窗（Dialog / AlertDialog）**：直接使用 shadcn/ui 组件，无需额外类名。通过 CSS 变量自动适配暗色主题。
2. **下拉菜单（DropdownMenu / Select）**：同上，`bg-popover text-popover-foreground` 自动生效。
3. **抽屉（Sheet）**：移动端侧边栏使用 `bg-sidebar` 而非 `bg-background`，保持与桌面端侧边栏一致。
4. **Toast（Sonner）**：Toaster 通过检测 `<html>` 上的 `dark` 类自动切换主题，不依赖 `next-themes`。
5. **Tooltip**：使用默认 `bg-foreground text-background`（暗色背景上的亮色提示气泡）。

#### 布局结构

```
┌──────────────────────────────────────────────┐
│ <html class="dark admin-dark">               │
│  ┌──────────┬───────────────────────────────┐ │
│  │ Sidebar  │ Topbar                        │ │
│  │ w-60     ├───────────────────────────────┤ │
│  │ (可折叠   │ Main content                  │ │
│  │  → w-16) │ p-4 md:p-6                    │ │
│  │          │                               │ │
│  └──────────┴───────────────────────────────┘ │
│  Portal 弹出层（Dialog 等）也在 <html> 内       │
└──────────────────────────────────────────────┘
```

- **侧边栏**：桌面端固定左侧 240px，可折叠至 64px；移动端使用 Sheet 抽屉
- **顶栏**：sticky 定位，高 56px，含移动端菜单按钮和用户下拉菜单
- **内容区**：`flex-1`，内边距 `p-4 md:p-6`

#### 科技感装饰元素

- Logo 徽章：`shadow-[0_0_12px_rgba(0,200,255,0.2)]` 青色微光
- 激活导航项：`bg-primary/15` 底色 + `shadow-[inset_0_0_0_1px_rgba(0,200,255,0.15)]` 内发光
- 激活图标：`drop-shadow-[0_0_4px_rgba(0,200,255,0.4)]` 光晕
- 卡片悬停：`hover:border-primary/30 hover:shadow-[0_0_20px_rgba(0,200,255,0.05)]`
- 登录按钮：`shadow-[0_0_20px_rgba(0,200,255,0.15)]` 微光
- 分组标题：`text-primary/60`，10px 大写字母间距

#### 后台页面模板

新增后台模块页面时遵循统一结构：

```tsx

  const data = await xxxService.getAll();

  return (
    <div>
      <div className="mb-6 flex items-center justify-between">
        <div>
          <h1 className="text-2xl font-bold">模块名称</h1>

```

#### 不允许的做法

- 在后台组件上手动加 `admin-dark` 类（主题已通过 `<html>` 全局生效）
- 在后台组件中使用硬编码颜色值（如 `text-white`、`bg-gray-900`），应使用语义色
- 在弹出组件上手动设置背景色（如 `className="bg-gray-800"`），应依赖 CSS 变量
- 使用 `next-themes` 的 `useTheme()` 或 `ThemeProvider`（后台主题通过 `useAdminTheme` hook 管理）

### 14.4 错误处理

- **Service 层**：抛出具体错误，使用自定义 Error 类
- **Action 层**：catch 所有错误，返回统一的 `ActionResult` 格式
- **UI 层**：根据 `ActionResult.success` 显示 toast 或错误信息

```typescript
// 自定义错误
export class NotFoundError extends Error {
  constructor(entity: string, id: string) {
    super(`${entity} not found: ${id}`);
  }
}

// Service 中抛出
throw new NotFoundError('Product', id);

// Action 中捕获
try {
  const result = await productService.deleteProduct(id);
  return { success: true, data: result };
} catch (error) {
  if (error instanceof NotFoundError) {
    return { success: false, error: 'Product not found' };
  }
  return { success: false, error: 'An unexpected error occurred' };
}
```

### 14.5 注释规范

- **不写显而易见的注释**（如 `// 获取产品` 后面跟 `getProduct()`）
- **复杂业务逻辑必须注释**（如翻译回退逻辑、URL 重定向规则）
- **所有 Service 函数写简短 JSDoc**：参数说明 + 返回值说明
- **所有数据库表在 Schema 文件顶部写一行注释说明用途**
