> **项目名称**：Vela（多语言展示型网站系统）  
> **文档性质**：产品功能规格，描述"做什么"而非"怎么做"  
> **配套文档**：`技术方案.md`（描述"怎么做"）  
> **读者**：AI 开发者、人类开发者、产品经理

## 一、这是什么产品

一套**纯展示型 B2B 外贸网站系统**。帮助制造业、贸易公司向全球客户展示产品、收集询盘线索。

**一句话定义**：多语言产品展示网站 + 询盘收集 + 可视化建站后台。

**类比**：它像一个"只展示不卖货的阿里巴巴国际站"，企业拥有自己的独立域名和品牌形象。

### 系统由两部分组成

| | 网站前端 | 运营后台 |
|--|---------|---------|
| **面向谁** | 全球潜在客户（访客） | 企业内部运营人员 |
| **作用** | 展示产品、提交询盘 | 管理内容、查看询盘、配置网站 |
| **访问方式** | 公开，无需登录 | 需登录，有角色权限 |
| **设备** | 移动端为主（60%+流量） | 桌面端为主 |

### 明确的边界

**做**：
- 产品展示（分类浏览、详情查看、搜索）
- 询盘收集（询盘篮、表单提交、邮件通知）
- 多语言支持（20-30 种语言，手动+自动翻译）
- SEO 优化（结构化数据、站点地图、多语言 URL）
- 可视化建站（拖拽页面区块、主题配置）

**不做**：
- 在线交易、支付、购物车
- 价格展示（B2B 价格因客户而异，线下报价）
- 库存管理
- CRM / 客户跟进（由 Salesforce、HubSpot 等专业工具负责）
- 在线客服 / 实时聊天（可通过脚本注入第三方工具）

## 二、贯穿全局的核心约束

以下约束影响系统中几乎每一个功能，必须在开发最初就融入骨架。

### 约束 1：零硬编码文本

**整个系统中不允许出现任何硬编码的用户可见文本。**

所有文本分为两类，存储方式不同但原则相同——前端只负责渲染，文字全部来自后端：

| 文本类型 | 举例 | 存储位置 |
|---------|------|---------|
| **UI 文本** | 按钮"Submit"、导航"Products"、提示"No results" | `ui_translations` 表 |
| **内容文本** | 产品名称、页面标题、分类描述 | 对应实体的翻译字段 |

```
❌ <button>View Details</button>
✅ <button>{t('product.viewDetails')}</button>

❌ <title>About Us</title>
✅ <title>{pageTranslation.title}</title>
```

**连错误提示、空状态文案、Cookie 弹窗文字也不例外。**

### 约束 2：每个可翻译实体都支持多语言

系统中以下实体都必须支持多语言翻译：
- 产品（名称、简介、详情、SEO）
- 产品参数（参数名、参数值）
- 分类（名称、描述、SEO）
- 页面（标题、SEO）
- 页面区块（标题、副标题、正文、按钮文字）
- 区块子项（标题、描述、正文）
- 新闻（标题、内容、摘要、SEO）
- 导航菜单（菜单名称）
- 全局配置（站点名、公司名、地址、页脚文字等）
- 询盘表单字段（标签、占位符、帮助文本）
- UI 翻译键值对

### 约束 3：语言回退机制

当某个实体缺少目标语言的翻译时，回退链为：
1. 精确匹配（如 `zh-CN`）
2. 语言前缀匹配（如 `zh-CN` 不存在则找 `zh`）
3. 默认语言（如 `en-US`）
4. 第一个可用翻译

### 约束 4：自动翻译能力

后台需要集成翻译引擎（Azure Translator / Google Translate），支持：
- 单条内容一键翻译（翻译一个产品到所有语言）
- 批量翻译（翻译所有产品到指定语言）
- 翻译后人工审核修改
- UI 翻译键的批量自动翻译

### 约束 5：URL 与多语言结构

**语言前缀规则**：默认语言无前缀，非默认语言带前缀。切换语言时保持当前页面，仅改变前缀。

**Slug 统一使用英文**，不做多语言 Slug。原因：全球通用、无 URL 编码问题、运营只需维护一套。

**分类嵌套只取最终分类 Slug**（不展开完整层级路径），保持 URL 简洁。

**完整 URL 路由表**：

```
默认语言                                      非默认语言（以 zh-CN 为例）
─────────────────────────────────────────     ──────────────────────────────────────────────
/                                             /zh-CN/
/products                                     /zh-CN/products
/products/[categorySlug]                      /zh-CN/products/[categorySlug]
/products/[categorySlug]/[productSlug]        /zh-CN/products/[categorySlug]/[productSlug]
/news                                         /zh-CN/news
/news/[slug]                                  /zh-CN/news/[slug]
/page/[slug]                                  /zh-CN/page/[slug]
/about                                        /zh-CN/about
/contact                                      /zh-CN/contact
/search                                       /zh-CN/search
```

**URL 规则说明**：

| 规则 | 说明 |
|------|------|
| 默认语言带前缀 → 301 重定向 | `/en-US/products` → 永久重定向到 `/products` |
| 分类嵌套 | 工业设备 > 水泵 > 离心泵，URL 只取 `/products/centrifugal-pumps`，不展开为 `/products/industrial-equipment/pumps/centrifugal-pumps` |
| 特殊页面 | `/about`、`/contact` 等有固定顶级 URL，对应后台 slug 为 `about`、`contact` 的页面 |
| 自定义页面 | 统一放在 `/page/[slug]` 下，如 `/page/factory-tour`、`/page/privacy-policy` |
| 询盘 | 无独立 URL，询盘表单以弹窗形式出现（从询盘篮或产品详情页触发） |
| Slug 格式 | 全部小写英文 + 连字符，如 `centrifugal-pump-x100`，不使用中文或其他语言字符 |

### 约束 6：移动端优先

60% 以上流量来自移动设备，且目标客户可能在弱网环境（东南亚、非洲、中东的 3G/4G）或通过社交媒体（WhatsApp 分享链接）打开页面。

**响应式断点**：

| 断点 | 范围 | 典型设备 |
|------|------|---------|
| 移动端 | 320px ~ 767px | iPhone SE ~ iPhone 15 Pro Max |
| 平板端 | 768px ~ 1024px | iPad Mini ~ iPad Pro |
| 桌面端 | 1025px ~ 1920px+ | 笔记本 ~ 外接显示器 |

**触控交互要求**：
- 所有可点击元素最小尺寸 44×44px
- 相邻可点击元素间距 ≥ 8px，防止误触
- 不依赖 hover 效果传递关键信息（移动端没有 hover），hover 只作为增强
- 弹窗和下拉菜单要足够大，手指可以轻松操作

**布局适配要求**：
- 桌面端多列网格（3~4 列）→ 移动端单列或双列
- 产品参数表 → 移动端横向可滚动，或改为竖向卡片排列
- 侧边栏分类导航 → 移动端变为顶部下拉或底部抽屉
- 图片画廊 → 移动端支持左右滑动手势
- 询盘篮弹窗 → 移动端变为全屏抽屉
- 搜索对话框 → 移动端变为全屏覆盖
- Header 导航 → 移动端变为汉堡菜单 + 侧边抽屉

**移动端性能要求**：
- 首屏加载 < 3 秒（3G 网络下）
- 图片必须提供多尺寸版本（移动端加载小图，桌面端加载大图）
- 首屏以外的图片全部懒加载
- 字体文件总大小 < 100KB（gzip），并配置系统字体回退
- 避免大型第三方脚本阻塞首屏渲染

### 约束 7：富文本内容规范

系统多处使用富文本（产品描述、页面区块正文、新闻内容等），需要兼顾编辑灵活性和自动翻译兼容性。

**支持的格式（翻译 API 安全）**：
- 标题（H2 ~ H4，H1 由页面标题占用）
- 加粗、斜体、下划线
- 有序列表、无序列表
- 链接（a 标签）
- 简单表格（2~4 列的参数/对比表）
- 引用块
- 图片插入（带 alt 文字）

**不支持（避免使用）**：
- 内嵌 iframe、视频播放器（用专门的视频区块替代）
- 自定义 HTML/CSS 类名
- 内嵌脚本
- 复杂嵌套表格

**粘贴行为**：从 Word、网页等外部来源粘贴内容时，编辑器自动清理格式，只保留上述支持的标签，去除多余的样式代码。

**与自动翻译的关系**：翻译 API（Azure/Google）原生支持 HTML 输入，会翻译文本节点并保留标签结构。但图片 alt 属性不会被自动翻译（已知限制，可接受）。

### 约束 8：RTL 语言支持

阿拉伯语、希伯来语等 RTL 语言需要：
- 整体布局镜像翻转
- 文字方向 right-to-left
- 语言表有 `rtl` 标记，前端根据当前语言设置 `dir="rtl"`

## 三、网站前端功能

### 3.1 全局布局

#### 顶部导航栏（Header）
- 公司 Logo（可配置，点击回首页）
- 导航菜单（从后台动态管理，支持多级下拉）
- 语言切换器（国旗 + 语言名，切换后保持当前页面）
- 搜索按钮（点击弹出搜索对话框）
- 询盘篮图标 + 数量角标
- 移动端：Logo + 汉堡菜单 + 询盘篮图标

导航菜单支持 4 种链接类型：
- 内部链接（如 /about）
- 外部链接（如 https://alibaba.com）
- 关联产品分类（自动生成分类下拉子菜单）
- 关联页面（如"关于我们"页面）

#### 页脚（Footer）
- 公司信息（名称、地址、联系方式）——全部从后台配置读取
- 社交媒体图标链接（Facebook、LinkedIn、YouTube 等）
- 快速导航链接
- 版权信息（多语言）
- WhatsApp 快速联系按钮

#### Cookie 同意弹窗
- 首次访问时显示
- 文案从翻译系统加载
- 接受/拒绝按钮
- 记住用户选择

### 3.2 首页

首页内容**完全由后台的页面区块系统驱动**，运营人员可自由组合以下区块：

- **Hero 首屏**：大图背景 + 标题 + 副标题 + CTA 按钮
- **轮播横幅**：多张图片自动轮播
- **产品展示**：自动拉取精选/最新/指定分类的产品卡片
- **分类导航**：产品分类入口网格
- **特性网格**：公司优势、服务特点（图标 + 文字）
- **数据统计**：关键数字动画展示（如"20 年经验"、"50 个国家"）
- **客户评价**：轮播或网格展示
- **合作伙伴 Logo 墙**
- **时间线**：公司发展历程
- **行动号召（CTA）**：引导客户发起询盘
- **富文本**：任意图文内容
- **双栏布局**：左图右文或右图左文

（区块系统详见第五章）

### 3.3 产品展示

#### 产品列表页（/products）

- 左侧：分类树形导航（可折叠，支持多级）
- 右侧：产品卡片网格
  - 产品图片
  - 产品名称（当前语言）
  - 简短描述
  - 标签（如"New"、"Hot"）
  - "添加到询盘篮"按钮
- 标签筛选栏（显示所有可用标签）
- 搜索框（按产品名称、SKU 搜索）
- 排序选项（最新、热门、名称）
- 分页（显示总数、当前页码）

#### 分类页面（/products/[categorySlug]）

- 与产品列表页类似，但显示当前分类及其子分类的产品，**以及附加分类包含该分类的产品**
- 面包屑导航（首页 > 产品 > 分类名）
- 分类可以有独立的模板配置（布局方式、每页数量、是否显示侧边栏）
- 分类页面可以有上下区块（类似首页的区块系统，用于 SEO 内容）

#### 产品详情页（/products/[categorySlug]/[productSlug]）

- **产品图片画廊**：
  - 缩略图列表 + 主图展示
  - 点击放大 / 全屏查看
  - 支持展示产品视频链接

- **产品基本信息**：
  - 产品名称（当前语言）
  - SKU 编号
  - 所属分类（可点击跳转）
  - 标签

- **产品详细描述**（富文本，当前语言）

- **产品参数表**：
  - 按分组展示（如"基本参数"、"技术参数"）
  - 每组是一个表格，两列：参数名 | 参数值
  - 参数名和值都是当前语言的翻译

- **操作按钮**：
  - "添加到询盘篮"（可选择数量）
  - "立即询盘"（打开快速询盘表单）

- **关联产品推荐**（同分类或手动关联的产品卡片）

- **面包屑导航**（首页 > 产品 > 分类 > 产品名）

- 产品详情页也可以有模板配置（布局方式、是否显示视频、是否显示参数表）

#### 产品搜索（/search）

- 搜索框 + 搜索结果列表
- 搜索范围：产品名称（所有语言）、SKU
- 搜索结果显示为产品卡片

### 3.4 询盘系统

#### 询盘篮

- 全局可用（任何页面都能添加产品）
- 数据持久化在浏览器 LocalStorage
- 功能：
  - 添加产品（图片、名称、SKU 自动带入）
  - 修改数量
  - 删除单个产品
  - 清空全部
  - 查看产品列表
- Header 上显示数量角标
- 刷新页面后数据保留
- 点击"提交询盘"进入询盘表单

#### 询盘表单（弹窗）

- 预设字段：
  - 姓名（必填）
  - 邮箱（必填）
  - 电话（可选）
  - 公司名称（可选）
  - 国家/地区（下拉选择，可选）
  - 留言（必填）
- 自定义字段（后台配置，动态渲染）
- 询盘篮中的产品清单自动带入
- 表单验证（前端即时验证 + 后端校验）
- 防垃圾：reCAPTCHA 或 hCaptcha
- 提交成功后显示感谢页面

#### 单产品快速询盘

- 产品详情页上的简化询盘表单
- 只需填写姓名、邮箱、留言
- 自动附带当前产品信息

#### 询盘邮件通知

提交询盘后触发两封邮件：

**1. 通知邮件（发给企业）**：
- 收件人：后台配置的通知邮箱（支持多个，如 sales@company.com, manager@company.com）
- 语言：英文
- 内容：客户姓名、邮箱、公司、国家、留言、产品清单（名称+SKU+数量）、提交时间、来源页面
- 邮件标题：`New Inquiry from {客户姓名} - {站点名称}`

**2. 确认邮件（发给客户）**：
- 收件人：客户填写的邮箱
- 语言：英文（固定）
- 内容：感谢提交、询盘编号、产品清单摘要、预计回复时间、公司联系方式
- 邮件标题：`Thank you for your inquiry - {站点名称}`

邮件发送依赖后台配置的 SMTP 服务（见 4.9 系统设置）。

### 3.5 内容页面

#### 自定义页面（/page/[slug]）

- 内容完全由后台的页面区块系统驱动
- 可以用来制作：关于我们、服务介绍、资质证书、工厂介绍等任意页面

#### 特殊页面

以下 URL 有固定路径，但内容从后台管理：

| URL | 用途 |
|-----|------|
| `/` | 首页 |
| `/about` | 关于我们 |
| `/contact` | 联系我们（包含联系表单、地图、联系方式） |
| `/products` | 产品中心 |
| `/news` | 新闻中心 |

#### 新闻（/news、/news/[slug]）

- 新闻列表页：封面图 + 标题 + 日期 + 摘要，按时间倒序
- 新闻详情页：封面图 + 标题 + 发布日期 + 正文（富文本）
- 面包屑导航

#### 联系我们（/contact）

- 联系信息（邮箱、电话、WhatsApp、地址等，从全局配置读取）
- 联系表单（姓名、邮箱、留言，提交到询盘系统）
- Google Maps 地图嵌入（经纬度或嵌入代码从后台配置）
- 社交媒体链接

### 3.6 SEO

SEO 是 B2B 外贸站的**核心获客渠道**（大部分询盘来自 Google 自然搜索），必须作为一等功能对待。

#### 3.6.1 页面级 SEO 元数据

每个可访问的 URL 都必须有完整的 SEO 元数据：

| 元数据 | 来源 | 回退 |
|--------|------|------|
| `<title>` | 实体的多语言 `seo_title` 字段 | 实体名称 + 站点名称 |
| `<meta description>` | 实体的多语言 `seo_description` 字段 | 实体的简短描述 |
| `<meta keywords>` | 全局配置的多语言关键词 | 无 |
| `<link rel="canonical">` | 当前页面的规范 URL | 自动生成 |
| Open Graph（og:title, og:description, og:image） | 同上 + 全局 OG 图片 | 自动从实体数据生成 |
| Twitter Card | 同 OG | 自动生成 |

**Title 格式规范**：
- 首页：`{站点名称}`
- 产品详情页：`{产品名称} | {站点名称}`
- 分类页：`{分类名称} | {站点名称}`
- 自定义页面：`{页面标题} | {站点名称}`
- 新闻详情：`{新闻标题} | {站点名称}`

以上所有文本都使用**当前语言**的翻译版本。

#### 3.6.2 多语言 SEO（关键）

**hreflang 标签**：

每个页面必须在 `<head>` 中输出所有语言版本的 hreflang 标签，告诉搜索引擎同一内容的不同语言版本之间的关系。

```html
<!-- 英文版产品详情页的 <head> 中 -->
<link rel="alternate" hreflang="en-US" href="https://example.com/products/pumps/pump-x100" />
<link rel="alternate" hreflang="zh-CN" href="https://example.com/zh-CN/products/pumps/pump-x100" />
<link rel="alternate" hreflang="es-ES" href="https://example.com/es-ES/products/pumps/pump-x100" />
<link rel="alternate" hreflang="x-default" href="https://example.com/products/pumps/pump-x100" />
```

规则：
- 为每个已启用的语言生成一条 hreflang
- `x-default` 指向默认语言版本（无前缀的 URL）
- hreflang 中的 URL 必须是绝对 URL（带域名）
- 所有语言版本页面的 hreflang 标签集必须完全一致（互相引用）

**canonical 标签**：

- 每个页面的 canonical 指向自身（当前语言版本的绝对 URL）
- 默认语言页面的 canonical 是无前缀 URL
- 非默认语言页面的 canonical 是带前缀 URL
- 不同语言版本的 canonical **不互相指向**（每个语言版本都是独立的规范页面）

**多语言 Sitemap**：

```xml
<url>
  <loc>https://example.com/products/pumps/pump-x100</loc>
  <xhtml:link rel="alternate" hreflang="en-US" href="https://example.com/products/pumps/pump-x100"/>
  <xhtml:link rel="alternate" hreflang="zh-CN" href="https://example.com/zh-CN/products/pumps/pump-x100"/>
  <xhtml:link rel="alternate" hreflang="es-ES" href="https://example.com/es-ES/products/pumps/pump-x100"/>
</url>
```

Sitemap 中每个 URL 条目都必须包含所有语言版本的 alternate 链接。

#### 3.6.3 结构化数据（Schema.org JSON-LD）

| 页面 | Schema 类型 | 关键字段 |
|------|------------|---------|
| 全站 | `Organization` | 公司名称、Logo、联系方式、社交链接 |
| 全站 | `WebSite` | 站点名称、URL、搜索功能 |
| 产品详情页 | `Product` | 产品名称、图片、SKU、描述、品牌 |
| 所有页面 | `BreadcrumbList` | 面包屑导航路径 |
| 新闻详情页 | `Article` | 标题、发布日期、作者、图片 |
| FAQ 区块 | `FAQPage` | 当页面包含 FAQ 区块时自动输出 |

注意：
- 结构化数据中的文本使用当前语言翻译
- `Product` Schema 中**不包含价格**（B2B 不展示价格）
- `Organization` Schema 全站统一，从全局配置读取
- JSON-LD 格式输出（不使用 Microdata 或 RDFa）

#### 3.6.4 Sitemap 与 robots.txt

**XML Sitemap**（自动生成）：
- 包含所有已发布的页面、产品、分类、新闻
- 每个 URL 包含所有语言版本的 alternate 链接
- 包含 `<lastmod>` 日期（实体的更新时间）
- 产品图片包含在 Sitemap 中（图片 Sitemap）
- Sitemap 地址写入 robots.txt

**robots.txt**：
```
User-agent: *
Allow: /
Disallow: /api/
Disallow: /search
Sitemap: https://example.com/sitemap.xml
```

`/search` 页面不索引（搜索结果页对 SEO 无价值）。

#### 3.6.5 其他 SEO 要求

- **URL 格式**：全小写、英文、连字符分隔，不含特殊字符（见约束 5）
- **图片 alt**：所有图片必须有 alt 属性，内容为当前语言的描述性文字
- **H1 唯一性**：每个页面有且仅有一个 H1 标签（页面标题或产品名称）
- **内部链接**：面包屑导航、关联产品、分类导航等自动生成内部链接
- **页面速度**：Lighthouse SEO 得分 > 90（见性能要求章节）
- **移动端友好**：所有页面通过 Google Mobile-Friendly Test（见约束 6）
- **无 404 死链**：产品/分类/页面下架后，如有旧 URL 被搜索引擎收录，应配置 301 重定向到分类页或首页

- 页面加载骨架屏（Skeleton）
- 404 自定义页面
- 500 错误页面
- 返回顶部按钮
- 面包屑导航（全站）
- 图片懒加载
- 页面切换平滑过渡

## 五、页面区块系统（详细）

这是网站最重要的功能之一。每个页面的内容由一组"区块"组成，运营人员可以自由添加、排序、配置区块来构建页面。

### 5.1 区块系统概念

区块系统采用**注册制**设计：不预设固定的区块列表，而是定义统一的区块协议（数据结构、渲染接口、开发规范）。任何符合协议的组件都可以作为区块注册到系统中，按需开发、即插即用。

**核心机制**：

- **区块注册表**：一个 `blockType → Component` 的映射，新增区块只需注册类型标识和对应组件
- **统一数据结构**：所有区块共享相同的数据库表结构和组件接口，区块内部只关心自己的渲染逻辑
- **通用配置外层处理**：背景、间距、动画等由外层容器统一渲染，区块不重复处理
- **按需开发**：根据业务需要随时扩展新区块，无需修改系统架构

**区块分类参考**：

| 类别 | 示例 | 特点 |
|------|------|------|
| 布局类 | Hero 首屏、双栏布局、间距、分隔线 | 控制页面结构和留白 |
| 内容类 | 富文本、特性网格、FAQ、数据统计、时间线、团队介绍、客户评价 | 展示图文信息 |
| 媒体类 | 轮播横幅、图片画廊、视频嵌入、合作伙伴 Logo | 展示多媒体内容 |
| 交互类 | 产品展示、分类导航、行动号召（CTA）、联系表单 | 数据驱动或用户交互 |
| 其他 | 自定义 HTML | 嵌入第三方代码 |

以上仅为分类参考和示例，实际区块类型不受限制。每个区块只需实现统一接口、遵守开发规范（见 5.5），即可无缝接入系统。

**区块是否包含子项**：

区块分为两种——**独立区块**和**子项区块**：
- **独立区块**（如 Hero、富文本、CTA）：自身内容由区块翻译字段承载，无子项
- **子项区块**（如特性网格、FAQ、轮播、时间线）：内容由一组可排序的子项承载，每个子项有独立的图标、图片、链接和多语言翻译

### 5.2 区块通用配置

每个区块都有以下通用配置项：

- **背景**：白色 / 灰色 / 主色 / 次色 / 渐变 / 背景图片
- **渐变**（当背景为渐变时）：预设方案（海洋蓝、日落橙、森林绿等）或自定义颜色 + 方向
- **背景图片叠加**：半透明遮罩层 + 透明度调节
- **上下间距**：无 / 小 / 标准 / 大 / 超大
- **容器宽度**：窄 / 默认 / 宽 / 全屏
- **入场动画**：无 / 淡入 / 上滑 / 下滑 / 缩放
- **锚点 ID**：用于页内跳转（如 #features）
- **自定义 CSS 类名**

### 5.3 区块子项

"有子项"的区块（如轮播、特性网格、FAQ）中，每个子项包含：

- 图标名称（Lucide 图标库）
- 图片（文件 ID）
- 链接 URL
- 自定义配置（JSON，区块特定）
- 多语言翻译：标题、描述、正文

### 5.4 区块多语言内容

每个区块自身有以下可翻译字段：
- 标题
- 副标题
- 正文内容（富文本）
- 按钮文字 + 按钮链接
- 次要按钮文字 + 次要按钮链接

### 5.5 区块开发标准（扩展新区块必须遵守）

本节定义所有区块的统一契约。**每个区块（无论是初期开发还是后续扩展）必须遵守以下全部规则**，以保证兼容性和一致性。

#### 规则 1：统一输入接口

每个区块组件接收完全相同的数据结构：

| 参数 | 内容 | 说明 |
|------|------|------|
| `section` | 区块数据 | 类型、配置 JSON、排序、状态 |
| `translation` | 当前语言翻译 | 已做过语言回退，区块直接使用即可 |
| `items` | 子项列表 | 每个子项已附带当前语言翻译 |
| `locale` | 当前语言代码 | 如 `zh-CN` |
| `defaultLocale` | 默认语言代码 | 如 `en-US` |

**区块本身不负责翻译查找和回退。** 翻译回退逻辑由统一的渲染调度器（SectionRenderer）在传入前完成。

#### 规则 2：通用配置由外层处理

所有区块共享以下通用配置，由外层容器（SectionWrapper）统一渲染，区块内部**不重复处理**：

- 背景色/背景图/渐变
- 背景图叠加遮罩
- 上下内边距
- 容器最大宽度
- 入场动画
- 锚点 ID
- 自定义 CSS 类名

区块只负责**内部内容区域的布局**。如果区块需要额外的特定配置（如轮播的自动播放间隔、网格的列数），放在 `config` 中扩展，字段命名使用 `snake_case`。

#### 规则 3：防御性渲染（必须处理空数据）

每个区块必须安全处理以下情况，**不得出现白屏、报错或显示 `undefined`**：

| 情况 | 处理方式 |
|------|---------|
| 翻译标题/副标题为空 | 不渲染该元素（而非显示空标签） |
| 翻译正文为空 | 不渲染正文区域 |
| 按钮文字为空 | 不显示按钮 |
| 图片字段为空 | 显示占位色块或隐藏图片区域 |
| 子项列表为空 | 隐藏整个区块，或显示空状态提示 |
| config 中某个字段缺失 | 使用默认值，每个配置字段必须定义默认值 |

#### 规则 4：响应式三断点

每个区块必须在以下三个断点下正常工作且视觉合理：

- **移动端（< 768px）**：通常单列，内容垂直堆叠
- **平板端（768px ~ 1024px）**：可双列或缩小间距
- **桌面端（> 1024px）**：完整布局

不允许出现：移动端内容溢出、横向滚动条、文字被截断、图片变形等问题。

#### 规则 5：不依赖 hover

区块中所有交互信息不能仅通过 hover 展示。Hover 效果只作为桌面端的视觉增强，移动端用户必须能在不 hover 的情况下获取所有信息和完成所有操作。

#### 规则 6：图片处理

- 所有图片使用统一的图片组件，自动处理多尺寸和格式转换
- 必须提供 `alt` 属性（从翻译或子项标题中获取）
- 非首屏图片使用懒加载
- 图片容器必须指定宽高比，避免布局偏移（CLS）

#### 规则 7：可独立测试

每个区块应该能独立渲染和测试，不依赖其他区块的存在。传入标准的 mock 数据即可渲染完整效果。

### 5.6 扩展新区块的步骤

新增一个区块只需完成以下步骤：

1. 确定区块类型标识（如 `feature_grid`），遵循 `snake_case` 命名
2. 确定是独立区块还是子项区块
3. 定义区块特定配置字段（如列数、样式变体），使用 JSON 存储
4. 开发前端渲染组件，实现统一接口（见规则 1）
5. 在区块注册表中注册 `type → Component` 映射
6. 开发后台编辑组件（配置表单 + 子项管理）

无需修改数据库结构、无需修改渲染调度器、无需修改通用配置逻辑。

## 六、多语言系统（详细）

### 6.1 语言管理

- 语言表字段：代码（主键，如 `en-US`）、英文名、本地名（如"简体中文"）、是否 RTL、排序、状态（active/inactive）、是否默认
- 运营人员可随时添加新语言，无需重新部署
- 前端语言切换器根据语言表动态生成

### 6.2 URL 与语言检测

完整的 URL 路由表和 Slug 规则见第二章"约束 5"。

语言检测与切换行为：
- 首次访问：中间件检测浏览器 `Accept-Language`，自动重定向到最佳匹配语言
- Cookie 记住用户语言偏好（有效期 1 年）
- 语言切换器切换后保持当前页面，同时更新 Cookie
- 爬虫（Googlebot 等）始终使用默认语言，不做重定向
- 默认语言 URL 如果带了前缀（如 `/en-US/products`），自动 301 重定向到无前缀版本

### 6.3 自动翻译

后台翻译管理模块支持调用翻译 API（Azure Translator）：

- **单条翻译**：在产品编辑页，点击"自动翻译到所有语言"
- **批量翻译**：选择实体类型 + 目标语言，一键翻译所有未翻译内容
- **UI 翻译批量翻译**：选择目标语言，翻译所有 UI 翻译键
- 翻译结果可预览、编辑后保存
- 翻译为异步执行，显示进度

### 6.4 翻译完成度

后台显示各实体的翻译完成度：
- 产品翻译完成度：85%（已翻译 170/200 条 × 10 种语言）
- 具体到每种语言的完成度
- 缺失翻译列表，点击可直接跳转编辑

## 八、典型用户旅程

### 旅程 A：海外采购商找到产品并提交询盘

```
1. Google 搜索 "industrial water pump manufacturer"
2. 点击搜索结果，进入产品分类页（英文版）
3. 浏览产品列表，点击感兴趣的产品
4. 查看产品图片、参数、描述
5. 点击"添加到询盘篮"
6. 继续浏览，又添加 2 个产品
7. 点击 Header 上的询盘篮图标
8. 查看已选的 3 个产品，点击"提交询盘"
9. 填写姓名、邮箱、公司、留言
10. 提交成功，看到感谢页面
11. 销售团队收到邮件通知
```

### 旅程 B：运营人员发布新产品

```
1. 登录后台
2. 进入"产品管理"，点击"新建产品"
3. 填写 SKU、选择分类、上传图片（5 张）
4. 填写中文产品名称、描述
5. 添加产品参数（功率、电压、材质等）
6. 点击"自动翻译到所有语言"
7. 审核英文、西班牙文翻译，微调后保存
8. 将状态改为"已发布"
9. 产品立即出现在网站上
```

### 旅程 C：运营人员用区块系统搭建"关于我们"页面

```
1. 登录后台 → 页面管理 → 新建页面
2. Slug 设为 "about"，模板选择 "full-width"
3. 添加区块：Hero 首屏（上传公司大楼照片，写标题"20 Years of Excellence"）
4. 添加区块：数据统计（4 个数字：成立年份、国家覆盖、产品数量、客户数量）
5. 添加区块：时间线（公司发展历程，每个时间节点是一个子项）
6. 添加区块：团队介绍（上传团队成员照片和介绍）
7. 添加区块：合作伙伴 Logo 墙
8. 添加区块：CTA 行动号召（"联系我们获取报价"）
9. 为每个区块填写中英文内容
10. 对所有区块点击"自动翻译"
11. 发布页面
```

## 十、适用行业

本系统是通用型 B2B 展示站，但以下行业特别适合：

- **机械制造**：工业泵、阀门、电机、设备
- **电子元器件**：传感器、连接器、电路板
- **建材五金**：管件、紧固件、工具
- **化工原料**：涂料、粘合剂、添加剂
- **家居用品**：家具、灯具、装饰品
- **汽车配件**：滤芯、刹车片、轴承
- **包装材料**：纸箱、塑料袋、标签

共同特点：产品有技术参数、需要多语言展示、依赖询盘获客。
