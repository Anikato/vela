## 重要决策记录

| 日期 | 决策 | 原因 |
|------|------|------|
| 2026-03 | 项目命名为 Vela | 拉丁语"帆"，契合全球贸易定位，简短好记 |
| 2026-03 | 采用单 Next.js 应用架构（不做前后端分离） | AI 开发效率优先，类型端到端贯通，Server Actions 替代 REST API |
| 2026-03 | 区块系统采用注册制，不预设固定数量 | 灵活扩展，按业务需要按需开发 |
| 2026-03 | 数据库选用 PostgreSQL + Drizzle ORM | JSON 字段支持区块 config，Drizzle 类型推导强，AI 友好 |
| 2026-03 | Docker 镜像改用 postgres:17-alpine | 本地已缓存该镜像，避免拉取超时 |
| 2026-03 | 后台 UI 固定使用中文 + 暗色科技感主题 | 后台面向内部使用，无需多语言；`.admin-dark` CSS 作用域 |
| 2026-03 | 后台暗色主题改为挂载到 `<html>` 而非内部 `<div>` | 解决 Radix Portal 弹出内容（Dialog/DropdownMenu/Toast 等）无法继承暗色 CSS 变量的问题；通过 `useAdminTheme()` hook 管理 |
| 2026-03 | Sonner Toast 移除 `next-themes` 依赖 | 改用 `useDetectTheme()` 检测 `<html>` 上的 `dark` 类，避免依赖缺失的 ThemeProvider |
| 2026-03 | db 脚本统一使用 dotenv-cli 加载 .env.local | drizzle-kit 不自动读取 Next.js 的 .env.local |
| 2026-03 | 后台导航按功能域分 5 组：内容/营销/外观/多语言/系统 | 清晰的信息架构，便于后续扩展 |
| 2026-03 | 询盘管理拆分为独立页面（列表/表单配置/统计） | 功能独立，侧边栏子菜单导航 |
| 2026-03 | 设置页面拆分为独立子页面 | 站点设置/邮件配置/脚本注入各自独立 |
| 2026-03 | 用户角色简化为仅 Admin | 当前阶段不需要多角色，简化开发 |
| 2026-03 | 开发端口改为 3100 | 避免与常用端口冲突 |
| 2026-03 | 语言代码采用 BCP 47 风格作为内部统一标准，并维护 Azure/Google 映射 | 避免翻译引擎代码差异导致歧义；支持标准中文语言名称展示（如“克罗地亚语”） |
| 2026-03 | 文件存储抽象为 StorageAdapter 并默认本地实现 | 为后续切换 S3/R2 保持接口稳定，上传入口统一通过 media service |
| 2026-03 | 建立 API 与 Action 双文档机制，并要求接口变更强制同步文档 | 降低前后端联调成本，避免接口契约漂移 |
| 2026-03 | API/Action 文档新增模板段与 Deprecated 规范 | 统一文档写法，保障接口演进可追踪、可迁移 |
| 2026-03 | 媒体库模块采用“上传 API + 删除 Action”混合模式 | 上传使用 multipart API，删除走 Server Action，兼顾文件传输和业务一致性 |
| 2026-03 | 用户管理在 P1 先实现单角色 Admin 能力 | 优先满足后台账号维护与安全控制，避免过早引入复杂 RBAC |
| 2026-03 | 富文本编辑器基于 Tiptap 并采用白名单 HTML 清理 | 满足翻译友好的基础格式能力，减少粘贴外部内容带来的脏标签风险 |
